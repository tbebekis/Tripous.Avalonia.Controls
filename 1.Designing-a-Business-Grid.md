# Designing a Business Grid

## 1. Introduction

A business grid, in its simplest form, looks like a data table. It has rows, columns, and cells. The user sees values, navigates, selects, changes data, and runs actions. That image, however, is deceptively simple. In business applications, the grid is not just a way to display data. It is the main point where the user meets the business model.

A business grid must present large amounts of data, organize it, filter it, group it, summarize it, edit it, and connect it with validation rules, permissions, commands, and workflows. For that reason, the right architectural approach is not to see it as one large control. It is more useful to see it as a small runtime framework.

That framework has its own state model, its own rendering pipeline, its own navigation model, its own editing engine, and clear collaboration points with the data source and the application.

This article describes the business grid as a general architectural concept. It is not tied to a specific framework, technology, or product. The goal is to explain why the core concepts of a serious grid exist and how they work together.

## Part I - The Problem

## 2. Why Business Applications Need Grids

### Why does this concept exist?

Business applications handle collections of similar objects: documents, customers, products, transactions, order lines, accounting entries, warehouses, tasks, requests, and payments. The user rarely works with a single object isolated from the rest. Usually the user needs to see lists, compare values, find exceptions, make bulk corrections, and understand the overall picture.

The grid exists because this work is naturally tabular. Business entities have fields. Their collections have rows. User decisions are based on comparing values by column and by row. A grid allows the user to scan visually, sort, filter, move quickly, and edit without losing context.

In many applications, the grid is the real workspace. It is not a helper element. It is where daily work happens.

### How does it work with the other concepts?

Because the grid is a workspace, it cannot be limited to display. It must work with:

- The data source, from which it reads values and to which it writes values back.
- The columns, which describe how each field is interpreted.
- The navigation model, which enables fast movement.
- The editing model, which organizes value changes.
- Filtering, sorting, and grouping, which turn raw data into a useful view.
- Summaries, which provide an overall picture.
- Commands, which connect the grid with business actions.

This makes the grid a collaboration hub. It does not own the business logic by itself, but it must support it in a clean and extensible way.

## 3. Why Traditional Table Controls Are Not Enough

### Why does this concept exist?

A simple table control usually answers the question: how do I display a collection of values in rows and columns? That is enough for simple lists. It is not enough for business work.

In a business grid, the user needs more:

- Change values inside the grid.
- See different editors per field type.
- Perform lookups in auxiliary tables.
- See aggregates.
- Filter with business terms.
- Group and analyze.
- Handle many thousands of rows.
- Keep context stable while navigating.
- See what is allowed and what is forbidden per column, row, or cell.

A generic table control usually does not have a rich enough runtime model for all this. If we try to add these features casually, the result becomes a set of special cases. The grid starts to behave unpredictably because there is no clear architecture.

### How does it work with the other concepts?

The limitations of a simple table lead to the need for separate subsystems. Display must be separated from the data model. Selection must be separated from the current cell. Editing must become a separate session. Sorting and filtering must belong to a view mechanism. Rendering must be based on virtualization instead of thousands of real controls.

In other words, the business grid exists because a simple representation of rows and columns is not enough. It needs a runtime that coordinates many concepts.

## Part II - The Runtime Model

## 4. The Grid as a Runtime System

### Why does this concept exist?

A business grid cannot be static. Its state changes constantly. The user navigates, selects, opens editors, closes editors, changes filters, sorts, groups, scrolls, adds rows, deletes rows, and changes the view.

This dynamic state needs a runtime model. The grid must know not only which data it displays, but also the state of that data's view.

Core runtime concepts include:

- Columns: the description of fields.
- Rows: the displayed rows of the view.
- Current Cell: the cell where the active work position is.
- Current Row: the active row.
- Selection: what the user has selected.
- Editing: whether there is an active edit session.
- Scrolling: which part of the view is visible.
- Groups: how rows are organized into a hierarchy.
- Filters: which subset of the data is displayed.
- Sorting: the order in which data is displayed.

Without a runtime model, every interaction would have to be calculated in isolation. That leads to inconsistencies: one cell looks current, another cell receives input, a different row is selected, and another row index is written back to the data source.

Columns are not just headers. They are descriptors. A base column defines the common characteristics: field name, title, width, visibility, formatting, alignment, editability, and the ability to participate in operations such as sorting, filtering, or summaries.

Specialized column types appear on top of that base concept:

- Text column.
- Number column.
- Date column.
- Boolean column.
- Lookup column.
- Custom business column.

The existence of column types is architecturally important. The grid should not constantly guess what a value means. The column is the place where knowledge about the field is concentrated. A number column knows how to parse numbers and which aggregates are valid. A date column knows how to display and read dates. A lookup column knows that the stored value may be a code, while the displayed value is a description.

### How does it work with the other concepts?

The runtime model is the coordination center. Columns describe what each cell means. The data source provides the values. The rendering pipeline asks the runtime model what it should display. The navigation manager changes the current cell. The selection manager changes the selection. The editing engine starts and completes edit sessions. The scroll manager determines which rows are visible.

The runtime model is not necessarily visible to the user, but every visible behavior passes through it.

A simple pseudo-model might look like this:

```text
GridRuntime
    Columns
        BaseColumn
        TextColumn
        NumberColumn
        DateColumn
        LookupColumn
    ViewRows
    CurrentCell
    Selection
    EditSession
    ScrollState
    SortState
    FilterState
    GroupState
```

The exact shape is not important. What matters is that the grid needs one coherent picture of its state.

## 5. The Data Model

### Why does this concept exist?

The grid presents data. We should not assume that it owns the data. The data belongs to an independent data source: a collection of objects, a tabular model, a remote source, a projection, or any other model that provides rows and values.

This separation is critical. If the grid stores the data as its own internal structure, it becomes difficult to cooperate with the application, support validation, transactions, lazy loading, permissions, undo, audit, synchronization, or server-side operations.

The grid must read and write through a clean contract. It does not need to know where the data really lives.

### How does it work with the other concepts?

The data source provides three basic capabilities:

- Read values.
- Write values.
- Provide information about whether a row or cell can change.

The grid uses these capabilities in all subsystems:

- The renderer asks for values to paint them.
- The editing engine asks for the old value and writes the new one.
- The filtering engine reads values to decide which rows pass.
- The sorting engine compares values.
- The grouping engine creates group keys.
- The summary engine calculates aggregates.

A generic data access contract can be expressed like this:

```text
DataSource
    RowCount
    GetValue(row, column)
    SetValue(row, column, value)
    CanEdit(row, column)
    Insert(row)
    Delete(row)
```

The grid does not need to know whether memory, a database, or a service is behind this contract. It only needs a stable collaboration contract.

## Part III - Rendering

## 6. Rendering vs Controls

### Why does this concept exist?

A business grid can display thousands or millions of cells. If every cell were a real UI element, the cost would be enormous. There would be thousands of objects, thousands of layout calculations, thousands of event routes, and high memory consumption.

That is why professional grids usually paint their content. The cell is not necessarily a control. It is a rectangle with text, colors, borders, icons, glyphs, and states. Real controls are created only when needed, mainly for editing.

This approach enables:

- High performance.
- Virtualization.
- Stable scrolling.
- Full control over appearance.
- A small number of real UI objects.

### How does it work with the other concepts?

The rendering pipeline works with the runtime model. It does not decide by itself what exists. It asks:

- Which columns are visible?
- Which rows are visible?
- Which cell is current?
- Which rows are selected?
- Which columns have a sorting glyph?
- Which filters are active?
- Which groups are expanded?
- Is there an active editor that must appear over a cell?

Rendering is not the data model. It is a view of state. If the runtime model is correct, rendering can be simple and predictable.

## 7. Coordinate System

### Why does this concept exist?

The grid must constantly translate between logical concepts and visual coordinates. The user does not click a row object. The user clicks a point on the screen. The grid must understand whether that point corresponds to a cell, header, filter cell, group row, scrollbar, resize handle, or empty space.

This requires a coordinate system.

Core concepts are:

- Row index.
- Column index.
- Cell rectangle.
- Visible area.
- Horizontal offset.
- Vertical offset.
- Hit testing.
- Scroll position.

Without a clear coordinate system, interactions become fragile. A click may select the wrong cell. An editor may appear in the wrong position. A drag operation may fail to find the correct drop target.

### How does it work with the other concepts?

The coordinate system is the bridge between rendering and interaction.

The renderer uses it to paint:

```text
for each visible row
    for each visible column
        rect = getCellRectangle(row, column)
        drawCell(rect, value)
```

Hit testing uses it in the opposite direction:

```text
hit = hitTest(pointerPosition)
if hit.kind == Cell
    setCurrentCell(hit.row, hit.column)
```

Scrolling changes the visible area. Navigation may ask for scrolling so the current cell becomes visible. The editing engine asks for the active cell rectangle so it can position the editor.

Coordinates are therefore not a display detail. They are a shared language for rendering, input, navigation, and editing.

## 8. Virtual Scrolling

### Why does this concept exist?

A business grid can have far more rows than fit on the screen. It may have ten thousand rows while the user sees only thirty. If the grid tries to create or paint all rows every time, performance collapses.

Virtual scrolling exists to solve this problem. The basic idea is that the grid does not treat the entire list as a visible surface. It treats the screen like an elevator cabin. Above the cabin there is a large invisible area. Below the cabin there is another large invisible area. The user feels as if they are moving through the whole building, but the grid paints only what is inside the cabin.

That cabin is the viewport.

```text
virtual space

    rows above viewport
    not rendered

    ┌─────────────────────┐
    │ viewport             │
    │ rendered rows only   │
    └─────────────────────┘

    rows below viewport
    not rendered
```

The scrollbar does not mean that there are real visual objects for all rows. It means that there is a total logical extent. The thumb shows where the viewport is within that extent.

### How does it work with the other concepts?

Virtual scrolling works with the coordinate system and the rendering pipeline. The grid must know:

- How many logical rows exist in total.
- Which row is the first visible row.
- How many rows fit in the viewport.
- Which vertical offset corresponds to the scrollbar position.
- Which cell rectangles exist only for the visible rows.

The general flow is:

```text
scroll position
    ↓
first visible row
    ↓
visible row range
    ↓
render only that range
```

Hit testing also works only inside the viewport. If the user clicks the third visible row, the grid translates that position into a logical row index:

```text
logicalRow = firstVisibleRow + visibleRowOffset
```

The editing engine needs the same model. An editor can appear only for a cell that is inside the viewport. If the user scrolls while there is an active editor, the grid must decide whether to move it, close it with commit, or cancel it. The decision is policy, but virtual scrolling makes the need clear: the editor is not part of all rows. It is a temporary element over the visible cell.

Virtual scrolling also works with grouping and filtering. The grid does not scroll over raw data rows. It scrolls over the current view. If there are groups, collapsed nodes, or filtered rows, the virtual space corresponds to the projection, not simply to the original data source.

## Part IV - User Interaction

## 9. Navigation

### Why does this concept exist?

The user of a business grid works quickly. They expect to move with arrow keys, Tab, Enter, Page Up, Page Down, Home, and End. They also expect the mouse to cooperate with the keyboard instead of creating separate state.

The navigation model exists to define how the active work position moves inside the grid. The central concept is the current cell. This is the point where the next action will be applied: edit, toggle, paste, command, or navigation.

Without a navigation model, every key handler would decide by itself what "right", "down", or "next editable cell" means. That leads to different behaviors depending on the entry point.

### How does it work with the other concepts?

The navigation manager works with:

- The column model, to know which columns are visible or editable.
- The row model, to know which rows are data rows and which are group or summary rows.
- The scroll manager, to keep the current cell visible.
- The editing engine, to decide what happens when the user presses Tab or Enter inside an editor.
- The selection manager, because changing the current cell often also updates selection.

Focus is also important. The grid must know when it has keyboard focus and when an active editor has focus. These two states cooperate, but they are not the same.

## 10. Selection

### Why does this concept exist?

Selection is not one concept. In a grid there are several related but different states:

- Current Cell: the active cell.
- Current Row: the row of the active cell.
- Focused Row: the row that has visual emphasis.
- Selected Rows: the rows selected for an action.
- Multiple Selection: more than one selected row.
- Block Selection: a rectangular range of cells.

These concepts are often confused. But confusion creates problems. A row may be current without being part of multi-selection. A cell may have focus while many rows are selected. A delete command may target selected rows, while an edit command targets the current cell.

### How does it work with the other concepts?

The selection manager works with navigation, rendering, and commands.

Navigation changes the current cell. Selection may follow it or may not. Rendering must show the current cell, current row, and selected rows differently. Commands must know which set their action targets.

For example:

```text
currentCell = row 10, column Amount
selectedRows = [10, 11, 12]

EditCommand -> uses currentCell
DeleteCommand -> uses selectedRows
CopyCommand -> may use selected block or selected rows
```

The architecture must allow these differences instead of hiding them.

## 11. Editing

### Why does this concept exist?

Editing in a business grid is not simply "a textbox inside the cell". It is a session. It has a beginning, intermediate state, and end.

An edit session must answer:

- Which cell are we editing?
- What was the original value?
- Which editor is needed?
- How is the text or selected value converted into the real value?
- When is commit performed?
- When is cancel performed?
- Who validates?
- What happens if the value is invalid?

Editing exists as a separate subsystem because it has its own lifecycle rules. It is not part of rendering, although it appears visually. It is not part of the data source, although it eventually writes to it. It is not simple navigation, although it is affected by Tab and Enter.

The general flow is:

```text
Begin Edit
    ↓
Create Editor
    ↓
Editor Host
    ↓
User Input
    ↓
Validate / Normalize
    ↓
Commit or Cancel
```

### How does it work with the other concepts?

The editing engine works with:

- The current cell, to know what it is editing.
- The column, to choose the editor and parser.
- The data source, to read the old value and write the new value.
- The validation layer, to accept or reject the change.
- The renderer, to show edit state.
- The coordinate system, to position the editor over the cell.
- The navigation model, for Tab, Enter, and Escape.

The editor is a temporary mechanism. It is created only when there is an active edit session and disappears when the session completes. This keeps the grid light and predictable.

Architecturally, it is useful to have a base in-place editor. This is not necessarily a simple text input. It is a small editor host with a common lifecycle:

- Load value.
- Focus.
- Select content, when that makes sense.
- Return current editor value.
- Commit request.
- Cancel request.
- Cleanup.

Specialized editors can be built on top of that base editor:

- Text editor.
- Number editor.
- Date editor.
- Boolean editor.
- Lookup editor.
- Custom business editor.

Inheritance here is not just technical convenience. It expresses an architectural contract: all editors participate in the same editing pipeline, while each one has a different internal mechanism.

Some editors are simple. A text editor may be only a text input. A number editor may be a text input with right alignment and numeric validation. Other editors are composite. A date editor may have a text input and a button that opens a calendar. A lookup editor may have a text display, a drop-down button, and a selection list. A custom editor may contain several internal controls.

What matters is that the grid should not confuse the editor with the cell. The cell is a logical position and a rendered rectangle. The editor is a temporary mechanism placed over the cell.

```text
InplaceEditorBase
    TextEditor
    NumberEditor
    DateEditor
        text area
        drop-down calendar
    LookupEditor
        display area
        drop-down list
    CustomEditor
```

The drop-down is also part of the architecture. It does not belong to the data source and it is not a permanent child of the grid. It is a temporary overlay that works with the editor session. It must know when it opens, when it receives focus, when it returns a value, and when it closes without commit.

## Part V - Data Operations

## 12. Sorting

### Why does this concept exist?

The user needs to change the order of data to find patterns. Which documents are most recent? Which customers have the highest balance? Which transactions have zero quantity? Sorting exists to change the view without necessarily changing the data itself.

In a business grid, sorting does not simply mean comparing strings. The column has a type. Values may be numbers, dates, lookup values, or calculated values. Sorting must respect the real value, not only the displayed text.

### How does it work with the other concepts?

The sorting engine works with:

- The column model, to know which column is sorted.
- The data source, to retrieve values.
- The view projection, to change the display order.
- Rendering, to show the sort glyph.
- Grouping, because sorting inside groups may mean something different from sorting the whole list.

Sorting must be state of the view. It should not necessarily reorder the underlying data source.

## 13. Filtering

### Why does this concept exist?

Filtering exists because the user rarely wants to see all data at once. The user wants to limit the view to what matters at that moment: a specific customer, date, status, amount, text, or combination of conditions.

A business filter must be richer than simple text search. It needs operators, wildcards, comparison semantics, and proper type handling.

Examples:

```text
> 100
<= 30
%abc
<> closed
```

### How does it work with the other concepts?

The filtering engine creates a subset of the view. This affects:

- The row count seen by the renderer.
- Navigation, because some rows disappear.
- Selection, because selected rows may no longer be visible.
- Grouping, because groups must be created over filtered rows.
- Summaries, because totals usually refer to the current view.

The filter is not just text in a filter cell. It is a rule that participates in the data projection.

## 14. Grouping

### Why does this concept exist?

Grouping exists to turn a flat list into an analytical structure. The user may want to see orders by customer, transactions by warehouse, expenses by category, or tasks by status.

Grouping is not simple sorting. It creates new rows that do not exist in the data source: group headers, group summaries, and perhaps collapsed placeholders. It therefore changes the shape of the view.

### How does it work with the other concepts?

The grouping engine works with:

- The grouped columns, which define the group keys.
- Sorting, because groups and their children have an order.
- Filtering, because groups contain only rows that pass the filters.
- The summary engine, to calculate aggregates per group.
- Rendering, to display group headers and expand/collapse glyphs.
- Navigation, because not all displayed rows are editable data rows.

The grid view stops being a simple list of data rows. It becomes a projection tree.

## 15. Summaries

### Why does this concept exist?

Business users need an overall picture. Seeing the rows is not enough. They want count, sum, min, max, average, and other aggregates. Summaries exist to quickly answer questions such as:

- How many rows are there?
- What is the total amount?
- What is the smallest or largest value?
- What is the average?

Summaries are especially important when they work with filtering and grouping. The same grid can show totals for the whole view and summaries for each group.

### How does it work with the other concepts?

The summary engine reads the current view, not necessarily all data. It works with:

- Filtering, to calculate over visible rows.
- Grouping, for group-level summaries.
- The column model, to know which aggregates are allowed per type.
- Rendering, to display footer summaries or group summaries.

An important architectural point is that not all aggregates are valid for all data types. Sum makes sense for numbers, not for dates or text. Count makes sense almost everywhere. Average may need a specific decision per type.

## Part VI - Architecture

## 16. Major Components

### Why does this concept exist?

A business grid is complex enough to require a clear internal division. If everything is placed in one large object, behavior becomes hard to understand and extend.

The main subsystems are:

- Grid: the public coordination point.
- Data Source: the contract for data access.
- Column: the description of field and behavior, often with specialized types.
- Row: the logical row of the view.
- Cell: the intersection of row and column.
- Renderer: the display.
- Editor: editing, usually through a base editor and specialized editors.
- Selection Manager: selection.
- Navigation Manager: navigation.
- Scroll Manager: the visible area and mapping of logical rows to viewport rows.

Columns and editors are often small type hierarchies. There is a base column that expresses the common contract of all fields and specialized columns that add type-specific behavior. Likewise, there is a base in-place editor that expresses the common lifecycle and specialized editors that change the internal UI or value selection mechanism.

The scroll manager is not simply an object that paints a scrollbar. It is the subsystem that maintains the relationship between the large logical view and the small visible viewport. Without it, the renderer does not know which rows to paint and hit testing cannot translate a visible row into a real logical row.

### How does it work with the other concepts?

The grid acts as an orchestrator. It does not necessarily do everything itself. It coordinates the subsystems.

A typical collaboration for a cell click is:

```text
Pointer Input
    ↓
Hit Testing
    ↓
Navigation Manager updates Current Cell
    ↓
Selection Manager updates Selection
    ↓
Renderer invalidates affected area
```

A typical collaboration for editing is:

```text
Command: Begin Edit
    ↓
Column selects Editor
    ↓
Editor Host positions editor
    ↓
Data Source provides old value
    ↓
User changes value
    ↓
Column parses value
    ↓
Validation accepts or rejects
    ↓
Data Source receives new value
```

The value of the architecture lies in clear collaboration, not in the number of classes.

## 17. The Rendering Pipeline

### Why does this concept exist?

The rendering pipeline exists to turn runtime state into an image. Without a pipeline, every change would lead to fragmented drawing. That makes consistency and performance difficult.

A grid has many bands and layers:

- Toolbar.
- Group panel.
- Column headers.
- Filter row.
- Body rows.
- Group rows.
- Summary rows.
- Scrollbars.
- Active editor.
- Drag indicators.

The rendering pipeline defines the order and rules of painting.

### How does it work with the other concepts?

The pipeline reads state from all other subsystems:

```text
Render
    draw background
    draw fixed bands
    draw headers
    draw filters
    draw visible rows
    draw summaries
    draw selection/current cell
    draw scrollbars
    draw transient overlays
```

The renderer should not change business state. It should mostly be a function of the current state. When navigation changes the current cell, the renderer only reflects it. When filtering changes view rows, the renderer only paints the new view.

## 18. The Editing Pipeline

### Why does this concept exist?

The editing pipeline exists because changing a value is a process, not a moment. From the first input to the final commit, the process includes editor selection, initial value, input, parsing, normalization, validation, and write-back.

If these are not organized into a pipeline, each editor behaves differently. The text editor commits one way, the numeric editor another, the date editor another, and the lookup editor another. That tires the user and makes the grid hard to maintain.

### How does it work with the other concepts?

The editing pipeline must be common, while editors may differ.

```text
BeginEdit(cell)
    if cell is editable
        create editor for column
        load current value
        position editor
        focus editor

CommitEdit()
    read editor value
    normalize value
    parse according to column
    validate
    write to data source
    close editor

CancelEdit()
    discard editor value
    restore visual state
    close editor
```

Editors are plugins inside the pipeline. They do not define the whole lifecycle by themselves. This lets the grid provide consistent behavior for Enter, Tab, Escape, focus loss, and validation.

## Part VII - Extensibility

## 19. Extension Points

### Why does this concept exist?

No business grid can predict every business need. One application may need a special lookup. Another may need a custom renderer for status cells. Another may need command buttons, custom aggregates, special hit testing, or a specialized editor.

Extension points exist to support these needs without breaking the core architecture.

Possible extension points:

- Editors.
- Painters.
- Commands.
- Custom rendering.
- Custom columns.
- Hit testing.
- Validation.
- Normalization.
- Aggregates.
- Toolbar actions.

### How does it work with the other concepts?

A good extension point does not bypass the runtime model. It works with it.

A custom editor must enter the editing pipeline. A custom painter must use the same cell rectangles. A custom command must know the current cell and selection. A custom column must provide format, parse, and type semantics.

Extension should feel like participation in the framework, not a patch outside it.

A generic example:

```text
Column
    FormatValue(value)
    ParseValue(editorValue)
    CreateEditor()
    CanAggregate(kind)
```

This is not an implementation for a specific technology. It is an architectural idea: the column is the place where knowledge about a field is concentrated.

## Part VIII - Conclusion

## 20. A Business Grid Is a Small Framework

### Why does this concept exist?

A business grid starts from a simple need: display rows and columns. Very quickly, however, it becomes something more. It must manage state, user input, views, transformations, editors, validation, scrolling, grouping, summaries, and commands.

That is why a business grid should not be designed as one large control with many special cases. It should be designed as a small framework.

It has:

- Runtime model.
- Rendering engine.
- Navigation engine.
- Selection model.
- Editing engine.
- Data source contract.
- Projection engine.
- Extension points.

### How does it work with the other concepts?

All concepts work around one common center: displaying business data in a way that is fast, consistent, and extensible.

The data source provides values. Columns give meaning to the values. The runtime model holds state. The renderer displays it. The navigation and selection model enable work. The editing engine enables changes. Sorting, filtering, grouping, and summaries turn data into knowledge. Extension points allow the application to adapt the grid to its own needs.

The success of a business grid is not judged only by whether it displays data. It is judged by whether it can become a reliable work environment. To achieve that, it needs architecture. And that architecture looks much more like a small runtime framework than a simple UI component.
