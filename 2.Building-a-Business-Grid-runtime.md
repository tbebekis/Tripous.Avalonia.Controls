# Building a Business Grid Runtime

## 1. Introduction

The previous step is to understand that a business grid is not simply a table. It is a small framework. The next step is to see how that framework can be built internally.

This article does not describe a specific implementation. It describes the way of thinking behind a grid runtime: what state it keeps, which subsystems cooperate, which pipelines are needed, and how design decisions keep the grid stable, fast, and extensible.

A business grid runtime must constantly answer questions such as:

- Which rows exist in the current view?
- Which of them are visible in the viewport?
- Which cell is current?
- Is there an active edit session?
- How is an editor value converted into a data source value?
- How does a filter affect grouping and summaries?
- What needs to be repainted after a change?

These questions are not answered correctly with scattered code. They need a runtime.

## Part I - Runtime Core

## 2. The Runtime State

### Why does this concept exist?

The grid is not a static image. It is live state. The user can navigate, select, edit, change sorting, apply filters, open groups, and scroll. If every subsystem keeps its own independent picture, the grid will start to have contradictions.

Runtime state exists so there is one central, coherent picture of the grid.

It typically includes:

- Columns.
- View rows.
- Current cell.
- Current row.
- Selection.
- Edit session.
- Scroll state.
- Sort state.
- Filter state.
- Group state.
- Summary state.

### How does it work with the other concepts?

Runtime state is the shared source of truth. The renderer reads it. Navigation changes it. Editing uses it to know which cell it is editing. The scroll manager updates it when the viewport changes. The projection engine rebuilds it when sorting, filtering, or grouping changes.

A simple model can look like this:

```text
GridRuntime
    DataAdapter
    Columns
    Projection
    CurrentCell
    Selection
    EditSession
    ScrollState
```

The exact shape is not important. What matters is that all decisions pass through a shared runtime model.

## 3. The Data Adapter Contract

### Why does this concept exist?

The grid should not know how data is stored. It may come from objects, a table, a service, a query, or a temporary cache. If the grid is tied to a specific data model, it loses its usefulness.

The data adapter exists to turn any data source into a stable contract that the grid understands.

### How does it work with the other concepts?

The adapter provides basic operations:

```text
RowCount
GetValue(row, column)
SetValue(row, column, value)
CanSetValue(row, column)
Insert(row)
Delete(row)
```

Rendering asks for values. Editing writes back. Filtering and sorting read values. Grouping creates keys. Summaries aggregate over values. All of these go through the adapter, not directly through the data source.

This keeps the grid independent of the data source type.

## 4. The View Projection

### Why does this concept exist?

Raw data rows are not always the rows that the user sees. If there is a filter, some rows are removed. If there is sorting, the order changes. If there is grouping, group rows are created that do not exist in the data source. If there are summaries, additional summary rows appear.

The view is a projection. It is not the raw data.

### How does it work with the other concepts?

The projection engine takes the data source and the current operation state:

```text
raw rows
    ↓ filtering
filtered rows
    ↓ sorting
sorted rows
    ↓ grouping
group tree
    ↓ flatten expanded nodes
view rows
```

The renderer does not paint raw rows. It paints view rows. Navigation does not move over raw rows. It moves over view rows. Virtual scrolling is also performed over the projection.

This separation is critical. It allows the grid to show complex organization without changing the data source.

## 5. Row Kinds

### Why does this concept exist?

In a simple table, every row is a data row. In a business grid this is not true. The visible view may contain different kinds of rows:

- Data row.
- Group header row.
- Group summary row.
- Footer summary row.
- Filter row.
- Placeholder row.

Each row kind has different behavior. Not all rows are editable. Not all rows participate in selection. Not all rows have the same cells. Not all rows are painted in the same way.

### How does it work with the other concepts?

The row kind affects navigation, rendering, editing, and hit testing.

For example:

```text
if row.kind == DataRow
    can edit cells

if row.kind == GroupHeader
    can expand or collapse

if row.kind == Summary
    display aggregates
```

The grid runtime must know what kind of row it is handling. Otherwise it will try to edit a summary row or ask a group header for a data value.

## Part II - Layout And Rendering

## 6. Layout Metrics

### Why does this concept exist?

The grid needs stable geometry. It must know heights, widths, offsets, areas, and scrollbars. If these are calculated casually in every place, rendering and hit testing will disagree.

Layout metrics exist to describe the physical shape of the grid.

### How does it work with the other concepts?

Typical metrics are:

- Toolbar height.
- Group panel height.
- Header height.
- Filter row height.
- Row height.
- Summary band height.
- Horizontal scrollbar height.
- Vertical scrollbar width.
- Column widths.
- Viewport rectangle.

The renderer uses metrics for drawing. Hit testing uses them to find what is under the pointer. The scroll manager uses them to calculate how many rows fit. The editor host uses them to position the editor over the cell.

## 7. Visible Range Calculation

### Why does this concept exist?

The grid should not paint all rows. It should paint only the rows inside the viewport. To do that, it must calculate the visible range of the projection.

This is the foundation of virtual scrolling.

### How does it work with the other concepts?

The calculation is roughly:

```text
firstVisibleRow = scrollOffset / rowHeight
visibleRowCount = viewportHeight / rowHeight
lastVisibleRow = firstVisibleRow + visibleRowCount
```

In practice there are details: partial rows, variable heights, group rows, summary rows, frozen bands. The basic idea remains the same. The grid calculates which part of the logical view fits inside the viewport cabin.

The renderer paints that range. Hit testing translates the Y coordinate into a visible row offset and then into a logical view row.

## 8. Cell Geometry

### Why does this concept exist?

Every cell must have a rectangle. Without cell geometry there can be no drawing, hit testing, editor placement, selection painting, or drag feedback.

### How does it work with the other concepts?

The cell rectangle comes from:

- The row position.
- The column position.
- The horizontal scroll offset.
- The vertical viewport offset.
- The column width.
- The row height.

```text
cellRect(row, column)
    x = column.left - horizontalOffset
    y = viewport.top + rowOffset * rowHeight
    width = column.width
    height = rowHeight
```

The same rectangle must be used by the renderer, hit tester, and editor host. If each subsystem calculates it differently, small but annoying inconsistencies appear.

## 9. Rendering Order

### Why does this concept exist?

The grid has many visual layers. If they are painted in the wrong order, the result will be muddy or incorrect. For example, selection must be behind the text but above the background. The current cell border must appear above the cell. The active editor must be above the painted cell.

### How does it work with the other concepts?

A typical rendering order is:

```text
draw background
draw fixed bands
draw headers
draw filter row
draw body rows
draw group rows
draw summary rows
draw selection
draw current cell
draw scrollbars
draw drag indicators
arrange active editor overlay
```

Rendering order is where runtime state becomes an image. It should not change state. It should represent state.

## 10. Hit Testing

### Why does this concept exist?

The user interacts with points on the screen. The grid must translate those points into logical concepts. That is what hit testing does.

### How does it work with the other concepts?

A hit result may contain:

- Band.
- Row kind.
- View row index.
- Data row index.
- Column.
- Cell rectangle.
- Hit kind.

Example:

```text
HitResult
    kind = Cell
    rowKind = DataRow
    viewRow = 42
    dataRow = 128
    column = Amount
```

Navigation uses the hit result for the current cell. Resize uses it for the column edge. Grouping uses it for drop targets. The context menu uses it to know which column or summary it refers to.

## Part III - Interaction

## 11. Navigation Engine

### Why does this concept exist?

Navigation must be consistent. The right arrow, Tab, Enter, and mouse click must end in predictable current-cell changes. If every input handler moves state by itself, the grid will have different rules per input.

### How does it work with the other concepts?

The navigation engine knows:

- Which columns are visible.
- Which rows are data rows.
- Which cells are editable.
- How to move to the next or previous cell.
- When to scroll so the current cell is visible.

```text
MoveNextCell()
    find next visible column
    skip non-data rows when needed
    update current cell
    scroll current cell into view
```

Navigation works closely with selection, editing, and scrolling.

## 12. Selection Model

### Why does this concept exist?

The current cell is not always the selection. The selected row is not always the current row. Multi-selection is not simply many current rows. These concepts need a separate model.

### How does it work with the other concepts?

The selection model is updated by mouse, keyboard, and commands. The renderer displays it. Commands use it to decide which rows they apply to.

```text
CurrentCell -> where keyboard action applies
SelectedRows -> where row command applies
SelectedBlock -> where copy/paste applies
```

A clean distinction avoids many mistakes in business workflows.

## 13. Scroll Engine

### Why does this concept exist?

The scroll engine is not just a scrollbar. It is the mechanism that moves the viewport over the logical projection. It is the grid's elevator.

Above the viewport there are logical rows that are not painted. Below the viewport there are also logical rows that are not painted. The grid must give the feeling that the user moves through the whole list, while in reality rendering happens only in the visible part.

### How does it work with the other concepts?

The scroll engine works with:

- Projection, for the total row count.
- Layout metrics, for viewport height.
- The renderer, for the visible range.
- Navigation, for scroll-to-current-cell.
- Hit testing, for mapping a visible row to a logical row.

```text
scrollOffset changes
    ↓
visible range changes
    ↓
renderer redraws visible rows
```

## 14. Drag Operations

### Why does this concept exist?

Business grids often allow structural changes by drag: resize columns, reorder columns, group columns, remove columns, move grouped fields. These actions are transient. They have a beginning, preview, and end.

### How does it work with the other concepts?

A drag operation needs:

- Start point.
- Current pointer position.
- Drag kind.
- Valid and invalid targets.
- Visual feedback.
- Commit action on release.

```text
BeginDrag(column)
    track pointer
    update drop target
    render ghost
    on release apply operation
```

Drag works with hit testing, rendering, the column model, and grouping.

## Part IV - Editing Runtime

## 15. Edit Session Lifecycle

### Why does this concept exist?

The edit session is the most important internal process of the grid. Displaying an editor is not enough. The grid must know that it is in editing mode and which cell the mode belongs to.

### How does it work with the other concepts?

The flow is:

```text
BeginEdit(cell)
    check editable
    store editing cell
    create editor
    load current value
    position editor
    focus editor

CommitEdit()
    read editor value
    normalize
    parse
    validate
    write to data source
    close editor

CancelEdit()
    discard editor value
    close editor
    restore grid focus
```

The editing lifecycle works with the current cell, column, data adapter, validation, renderer, and navigation.

## 16. Editor Factory

### Why does this concept exist?

There is no single editor for all cells. The grid needs a mechanism for choosing an editor per column, data type, or application rule.

### How does it work with the other concepts?

The column is the natural decision point. A number column gives a number editor. A date column gives a date editor. A lookup column gives a lookup editor. The application can intervene and provide a custom editor.

```text
CreateEditor(column)
    if application provides custom editor
        return custom editor
    if column is Number
        return NumberEditor
    if column is Date
        return DateEditor
    if column is Lookup
        return LookupEditor
    return TextEditor
```

The editor factory keeps the grid extensible without filling the editing engine with special cases.

## 17. Typed Editors

### Why does this concept exist?

Each data type has a different editing experience. Text needs free typing. Numbers need right alignment and numeric validation. Dates need normalization and perhaps a calendar. Lookups need selection from a list.

### How does it work with the other concepts?

All typed editors inherit from a common base editor or implement a common contract. That lets the grid handle them uniformly:

```text
editor.Value
editor.Focus()
editor.CommitRequested
editor.CancelRequested
editor.DropDownRequested
editor.Cleanup()
```

An editor may be simple or composite internally. The grid cares about the contract, not the internal details.

## 18. Value Pipeline

### Why does this concept exist?

The value returned by the editor is not always ready for the data source. It may be text that must become a number. It may be partial date input. It may be a selected lookup item that must become a foreign key.

The value pipeline exists to convert the editor value into the final data value.

### How does it work with the other concepts?

The flow is:

```text
editor value
    ↓ normalize
normalized value
    ↓ parse by column
typed value
    ↓ validate
accepted value
    ↓ write to data source
```

Normalization may be generic or custom. Parsing often belongs to the column. Validation may belong to the application or the business layer. The grid coordinates the process.

## 19. Drop-down Editors

### Why does this concept exist?

Some editors cannot be expressed well with a simple text input. A date may need a calendar. A lookup may need a selection list. A complex field may need a custom picker.

The drop-down exists to provide extra selection surface without turning the cell into a permanent control.

### How does it work with the other concepts?

The drop-down works with the edit session:

- It opens from an editor command.
- It is positioned relative to the edited cell.
- It can receive focus.
- It returns a value to the editor.
- It closes with commit or cancel.

```text
Editor requests drop-down
    ↓
Grid creates drop-down host
    ↓
User selects value
    ↓
Editor receives value
    ↓
Commit pipeline continues
```

The drop-down is not an independent business-logic window. It is part of the editing pipeline.

## Part V - Data Operations

## 20. Sorting Pipeline

### Why does this concept exist?

Sorting changes the order of the view. It must be based on real values and column types, not only on displayed text.

### How does it work with the other concepts?

The sorting pipeline uses sort descriptors:

```text
SortDescriptor
    column
    direction
    comparer
```

The projection engine applies descriptors before grouping or inside groups, depending on policy. The renderer displays sort indicators. Navigation continues to work over the new order.

## 21. Filtering Pipeline

### Why does this concept exist?

Filtering turns user input into a row-selection rule. The text typed by the user is not yet a predicate. It must become a filter expression.

### How does it work with the other concepts?

The flow is:

```text
filter text
    ↓ parse
filter expression
    ↓ compile/evaluate
predicate
    ↓ apply to rows
filtered projection
```

Filtering affects row count, grouping, summaries, selection, and navigation. That is why it must belong to the projection pipeline.

## 22. Grouping Pipeline

### Why does this concept exist?

Grouping creates a hierarchical view from flat rows. It is not simple display. It is a new structure.

### How does it work with the other concepts?

The flow is:

```text
filtered and sorted rows
    ↓ group by first descriptor
groups
    ↓ group by next descriptor
nested groups
    ↓ flatten expanded nodes
view rows
```

Grouping produces group headers, group summaries, and data rows. The renderer must know the row kind. Navigation must know which rows are editable. The summary engine calculates aggregates per group.

## 23. Summary Pipeline

### Why does this concept exist?

Summaries answer business questions. How many? What total? What minimum? What maximum? What average?

### How does it work with the other concepts?

The summary pipeline uses aggregate descriptors:

```text
SummaryDescriptor
    column
    aggregateKind
    scope
```

The scope may be group or total. The aggregate must be valid for the column type. The result is displayed by the renderer but calculated by the runtime over the current view.

## Part VI - Extensibility And API

## 24. Public API Surface

### Why does this concept exist?

A grid runtime has many internal details. They should not all become public API. A large and vague API makes the grid hard to use and hard to evolve.

### How does it work with the other concepts?

The public API should expose concepts that make sense to the application:

- Columns.
- Data source.
- Current cell.
- Selection.
- Commands.
- Events.
- Settings.
- Extension points.

Internal projection structures, layout cache, and rendering state can remain private. The application should cooperate with the grid, not enter its engine.

## 25. Events vs Overrides vs Services

### Why does this concept exist?

Extension can happen in many ways. If everything is events, the application can do a lot but cannot easily replace behavior. If everything is overrides, extension requires subclassing. If everything is services, the architecture can become heavy.

### How does it work with the other concepts?

A practical separation:

- Events for application decisions.
- Overrides for specialized grid variants.
- Services for replacing whole subsystems.

Example:

```text
Need to validate a value -> event
Need a custom editor for one column -> event or column factory
Need completely different rendering policy -> renderer service
```

The goal is to allow extension without breaking the runtime model.

## 26. Custom Columns

### Why does this concept exist?

Business applications have special fields. Text, number, and date are not always enough. There may be a status column, progress column, lookup column, command column, or calculated column.

### How does it work with the other concepts?

A custom column must participate in the contract:

```text
CustomColumn
    FormatValue(value)
    ParseValue(editorValue)
    CanAggregate(kind)
    CreateEditor()
    CompareValues(a, b)
```

This means the column is not just display. It is type-specific behavior that works with rendering, editing, sorting, filtering, and summaries.

## 27. Custom Editors

### Why does this concept exist?

No editor library covers every need. A user may need an editor with lookup, validation, multi-field input, or a special picker.

### How does it work with the other concepts?

The custom editor must enter the same lifecycle:

```text
load value
focus
return value
request commit
request cancel
cleanup
```

If it follows the contract, the grid does not need to know its internal details. The edit session remains unified.

## 28. Custom Commands And Toolbar

### Why does this concept exist?

A business grid is not only display and edit. It often has actions: insert, delete, edit, refresh, export, approve, post, cancel, and custom business commands.

### How does it work with the other concepts?

Commands must know the runtime context:

- Current row.
- Current cell.
- Selection.
- Data source capabilities.
- Edit state.

A command should not search inside the UI by itself. It should receive clean context from the grid runtime.

## Part VII - Reliability

## 29. Invalid State Prevention

### Why does this concept exist?

The hardest grid bugs are state bugs. An editor open on a cell that is no longer visible. Current cell in a hidden column. Selection on a row removed by a filter. A summary calculated before grouping has finished.

Preventing invalid state is a core responsibility of the runtime.

### How does it work with the other concepts?

Every change should pass through a small number of central methods:

```text
SetCurrentCell()
SetSelection()
BeginEdit()
CommitEdit()
CancelEdit()
RebuildProjection()
SetScrollOffset()
```

If all subsystems change state directly, inconsistencies are unavoidable. If the runtime controls transitions, the grid remains stable.

## 30. Performance Rules

### Why does this concept exist?

A business grid may be used all day. It is not enough for it to work correctly. It must work quickly and steadily.

### How does it work with the other concepts?

Basic rules:

- Do not create a real control for every cell.
- Paint only the viewport.
- Keep a projection cache.
- Rebuild projection only when needed.
- Do not remeasure layout without reason.
- Do not repaint the whole grid for a small change if invalidation can be limited.
- Close or cancel active editors when the underlying context is no longer valid.

Performance is not a final optimization. It is an architectural decision.

## 31. Testing The Runtime

### Why does this concept exist?

Rendering is hard to test completely, but the runtime can be tested very well. And it is worth testing, because the rules live there.

### How does it work with the other concepts?

Good test targets:

- Filtering expressions.
- Sorting comparers.
- Group projection.
- Summary calculations.
- Navigation rules.
- Edit commit/cancel.
- Date normalization.
- Numeric parsing.
- Selection transitions.
- Scroll visible range.

If these are correct, the visual layer becomes more reliable because it stands on a stable engine.

## 32. Conclusion

A business grid runtime is a small application engine. It is not simply a renderer and it is not simply a data viewer. It is a system that coordinates data access, projection, layout, rendering, interaction, editing, and extension points.

The core idea is not to let behaviors spread randomly. The grid needs central runtime state, a clean data adapter, a projection pipeline, a coordinate system, virtual scrolling, typed columns, typed editors, and a clear editing lifecycle.

When these exist, the grid can grow without becoming fragile. It can accept custom columns, custom editors, filtering, grouping, summaries, and commands without losing coherence.

This is the boundary between a simple table and a real business grid runtime.
